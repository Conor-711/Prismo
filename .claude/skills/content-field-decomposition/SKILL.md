# 内容→原子字段 拆解法（Content → Atomic Fields）

## 这是什么 · 心智模型

一个**内容产品**本质是「**内容单元**的集合 + 围绕它的筛选/分类/排序/展示」。要把这样的模块做好，
真正的工程难点不是画 UI，而是**确定字段**：把一团模糊的「内容」变成一组**有限的、有类型的、用户关心的字段**。
字段定对了，筛选器、分类标签、排序键、图表编码、徽章都是它的自然投影；字段错了，后面全是补丁。

**核心方法（一句话）**：拿一个内容单元，**沿用户的感知**把它递归拆成越来越小的部分，
直到每个叶子都是一个**目标用户真正会据以筛选/判断/排序的原子事实**——这些叶子就是该模块的最终字段。
这是工程师思维：把「内容」这个大字段，一路拆到不能再拆的小字段。

> 关键过滤器：**每个叶子字段都要能回答「目标用户会用它来筛选、排序或下判断吗？」**。
> 答案为否就砍掉。正是这个「用户关心」约束，让 schema 收敛、不爆炸。先明确**目标用户是谁、ta 在做什么决策**，
> 一切字段都服务于那个决策。

## 拆解流程（照着做）

**1. 锁定内容单元与目标用户。** 这个模块在反复展示什么「一条」东西？（一条观点帖 / 一个视频 / 一条评论 /
一个房源）。用户是谁、在做什么决策？字段的去留全部以「**对这个用户的这个决策有没有用**」为准绳。

**2. 一级拆解：按「感知部分」切。** 用户看一条内容时，本能会分辨几块：
**谁说的（作者）· 在哪说的（来源）· 多少人响应（互动）· 说了什么（内容本身）**。
这套「作者/来源/互动/内容」对绝大多数 UGC / 观点类内容都成立；换域时替换成该域的感知部分
（房源＝位置/价格/房屋本身/房东；商品＝卖家/平台/销量评价/商品本身）。

**3. 二级拆解：把每个部分拆到原子字段。** 对每一块继续问「用户关心它的哪些方面」。三条**反复出现的拆解原语**
帮你拆得干净（见下「拆解原语」）。拆到「不能再拆且用户仍关心」为止。

**4. 给每个叶子字段配元数据。** 至少记四列：
   - **取值/类型**：枚举？数值？0-100 分？多标签数组？双语文本？
   - **来源**：原始抓取 / 解析得到 / **AI 分类** / 计算派生。这决定成本与管线。
   - **用途**：filter（筛）/ sort（排）/ group（分组）/ badge（标签）/ encode（图表视觉编码）/ display（仅展示）。
     一个叶子若四样都不是，多半该砍。
   - **现状**：映射到既有实现时，标 ✅ 已有 / ⚠ 派生未存 / ❌ 缺失（gap）。缺口就是 backlog。

**5. 输出字段表 + 缺口清单。** 用下方「输出格式」。最后回看：哪些字段用户最关心却还没有？那是下一步要做的。

## 三条拆解原语（让方法可迁移的关键）

不知道某一块怎么往下拆时，套这三条：

**① 绝对值 vs. 同侪排名（absolute vs. ranked-in-peer-group）。** 凡是「量」（互动数、知名度、销量、价格），
都要拆成**两个**字段：原始绝对值，**和**在某个可比群体内的相对排名/分位。原因：跨群体的原始数字不可比
——X 的 1 万赞 ≠ Reddit 的 1 万 upvote ≠ 雪球的 1 万。用户真正关心的往往是「**在它那一类里算高还是低**」，
所以「按平台的排名/分位」常常比绝对值更有用。每见一个量，都问一句「**和谁比？**」。

**② 内容沿「领域决策框架 + 通用内容轴」拆。** 「内容本身」最难拆，分两组：
   - **领域决策框架**：该领域的人**做决策时脑子里的那几个维度**。投资＝估值 / 业务与成长 / 管理层 /
     竞争格局 / 宏观 / 催化剂 / 资金与盘面。换域换框架（餐厅＝口味/环境/服务/价格/卫生；论文＝问题/方法/数据/结论）。
     这组通常要**靠 AI 分类**打成多标签。
   - **通用内容轴**：几乎任何观点内容都有的几条正交轴——**语言**、**立场/情绪**（看多·中性·看空）、
     **时间视野/周期**（短·中·长线）、**与主题的相关度/聚焦度**（整篇在讲它，还是顺带提一句）。
     这几条是强力的筛选/排序维度，别漏。

**③ 区分「事实层」与「提炼层」。** 同一块内容有两种字段：**原始事实**（原文、原话 quote、链接、时间戳——
忠实、可追溯、建立可信度）与 **AI 提炼**（为什么看多的一句话、2-3 个要点、分类标签、相关度分——
便于筛选/概览）。两层都要、但要**分开存**：提炼用于索引和筛选，原文用于展示和取信。
理念：**展示原文、不蒸馏掉事实；AI 只做分类/翻译/打分这类「索引」活**。

## 输出格式

先给**拆解树**（让结构一目了然），再给**字段表**，最后给**缺口清单**。

### 1）拆解树
```
内容单元：<一条 X>
├─ 作者 ── 绝对知名度 · 平台内知名度排名 · 头像 · handle
├─ 来源 ── 平台枚举
├─ 互动 ── 互动数绝对值 · 平台内互动排名
└─ 内容 ── [领域框架] 估值/成长/… · [通用轴] 语言/立场/周期/相关度 · [事实] 原文/原话/链接 · [提炼] 理由/要点
```

### 2）字段表（每个叶子一行）
| 字段 key | 归属部分 | 取值/类型 | 来源 | 用途 | 现状 |
|---|---|---|---|---|---|
| `stance` | 内容·立场 | bull\|neutral\|bear | AI 分类 | filter/encode | ✅ |
| `relevance` | 内容·相关度 | 0-100 | AI 打分 | filter/sort | ✅ |
| `author_rank` | 作者·知名度排名 | 1..N（按平台） | 计算派生 | sort/badge | ❌ |

### 3）缺口清单
逐条列「用户关心但当前没有」的字段，附**为什么用户关心**与**怎么来（来源/管线）**。这是直接可排期的 backlog。

## 应用到「一个 UI 模块」时的额外动作

模块（尤其图表）不是字段的平铺，而是把字段**映射到视觉编码与交互**。拆完字段后，再做两步：

- **字段 → 视觉/交互映射**：每个轴/坐标/颜色/大小/分组/筛选器**背后是哪个字段**？反过来，**每个用户关心的字段，
  在这个模块里有没有被某个编码或筛选器用上**？没被用上的高价值字段，就是这个模块「还没表达出来」的信息。
- **盘点已有 vs. 缺口**：对照真实数据模型（models / 取数层 / 组件 props），逐字段标 ✅/⚠/❌。
  ⚠（派生未存）和 ❌（缺失）合起来就是该模块的演进路线。

## 心法

- **拆到「用户会据以行动」为止，不为拆而拆。** 叶子的检验是「能不能成为一个筛选器/排序键/标签」。
- **每个量都配一个『同侪排名』。** 这是最常被漏、却最提升体验的一类字段。
- **事实与提炼分开存。** 别让 AI 摘要顶掉可追溯的原文。
- **缺口即产品。** 拆解的最大价值不是确认已有字段，而是照出「用户最关心却还没做」的那几个。
